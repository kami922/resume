           Title: How I Built the -strings Feature for Mandiant’s GoReSym
           Go binaries are notoriously difficult to reverse engineer. Between the unique
           runtime structures and the way the pclntab handles metadata, extracting
           meaningful data from a compiled Go executable requires more than a standard
           strings utility.

           Recently I added a -strings feature to Mandiant's GoReSym. The PR added around
           3,000 lines total,but only ~1,000 is actual code. The other 2,000 are test and
           reference files used to validate our output against FLOSS, Mandiant's own string
           extraction tool. That validation is the interesting part and it's what caught four
           bugs in our first attempt.

           before Explaining this commit/changes , two concepts to understand first.


           in C, a string is bytes ending in \x00 .
           In Go, a string is a struct with two fields:
            type string struct {
                 ptr uintptr    // pointer to where the characters live in memory
                 len int        // how many bytes
           }

           So in a compiled Go binary, wherever string metadata lives, you see pairs of
           values: (pointer, length) . That's a StringCandidate .
            type StringCandidate struct {
                 Pointer uint64    // e.g. 0x406080   → points to "runtime.error" in .rodata
                 Length   uint64   // e.g. 13         → len("runtime.error")
           }

           extractStrings()
           You can't just scan for null-terminated strings like the standard strings tool does
           .Go strings aren't null-terminated. They're stored as (pointer, length) pairs pointing
           into a blob of characters.
           This functions extracts hundreds of strings like "bool", "runtime.error", "invalid
           memory address" from go binary.
           This function concludes itself in 5 steps.
           Step 1.Get the Sections.
           A binary is divided into sections. .text has machine code. .rodata has read-only
           data (where the actual string characters live). .data has the pointer+length pairs.
           We load all sections into memory.
           Step 2 Check for Binary type
           Is it 64-bit or 32-bit? Little-endian or big-endian? This tells us how many bytes to
           read for each pointer and in what byte order. A 64-bit Windows binary uses 8-
           byte pointers. A 32-bit Linux binary uses 4-byte pointers.
           Step 3.Scan for candidates
           it is findStringCandidates() function I have explain it below as well in its part.

GoReSym’blog Strings                                                                                4
           We walk through every data section byte by byte (in pointer-sized steps). At every
           position we read two values — a pointer and a length. We ask: could this be a real
           Go string struct?
           If pointer is 0 → garbage, skip
           If length is 0 → garbage, skip
           If length is 100,000+ → garbage, skip
           Everything that passes becomes a candidate. After scanning the whole binary we
           might have 8,000 candidates. Most are garbage,random bytes that happened to
           look like a pointer+length pair.


           Step 4. Find the Real String Table
           Here's the key insight: Go's compiler stores ALL strings sorted by length, shortest
           first, in one contiguous block. So the pointer+length pairs for the real strings,
           when read in order, have lengths that go: 4, 4, 4, 5, 5, 6, 7, 8... — never
           decreasing. Hundreds of them in a row.
           Random garbage candidates don't have this property. Their lengths jump around
           randomly — 4, 9, 2, 15, 1... The run breaks after 2-3 entries.
           So we sort all 8,000 candidates by length and find the longest sequence where
           lengths never decrease. The real string table produces a run of 600+ entries.
           Noise produces runs of 2-3. We take the winner.
           Step 5 — Read the actual characters
           Now we have 648 winning candidates. Each one says: "there's a string at address
           0x406080 , length 4 ." We go to that address in the binary, read 4 bytes, get "bool" .

           Validate it's real UTF-8 text. Check it's printable. Keep it.
           Repeat 648 times. Done.


           findStringCandidates()
           This function receives bytes from a section like .data and walks through it looking
           for pairs of numbers that could be a (pointer, length) string struct.



GoReSym’blog Strings                                                                                5
           It doesn't know which pairs are real. It collects everything that isn't obviously
           garbage. That's why the output is called candidates suspects, not confirmed
           strings.
           let’s suppose it receives below arguments
            data        = raw bytes of the .data section
           baseAddr    = 0x6A0000   (where this section starts in memory)
           is64bit   = true
           isLittleEndian = true




           it walks through the bytes in steps of 8 on 64-bit (or 4 on 32-bit) — not byte by
           byte. The reason is alignment: Go string structs always sit at pointer-aligned
           addresses. Stepping by 1 would read overlapping garbage values.
           At each position it does two things:
                 reads bytes in the correct order based on endianness (little-endian → least
                 significant byte first) to get the actual pointer and length values
                 rejects obvious garbage: ptr == 0 (null pointer), length == 0 (empty
                 string), length > 100000 (no real Go string is 100KB long)
           Everything that passes gets added to a slice called candidates and returned.


           Concept: Endianness
           Endianness is just which end of a number gets stored first in memory. Here we are
           checking for little endianess which means least significant byte first.
           The number `0x0400` stored in memory can be laid out two ways:
                 Little-endian (x86, ARM): `00 04` — least significant byte first
                 Big-endian (MIPS, PowerPC): `04 00` — most significant byte first
           If you read the bytes in the wrong order you get a completely wrong number.


           in summary.
           Walk through the raw bytes of a section in pointer-sized steps, read every pair of
           numbers as a (pointer, length) , reject obvious garbage, and return everything else
           as candidates.

           findLongestMonotonicRun()
           The single most important function in PR or the heart of it.

           The problem we have is we get candidates from findStringCandidates, it still
           contains garbage values. findStringCandidates removes the obviously fake IDs, The
           ones with no photo, no name, expired dates. But thousands of fake IDs still look
           real on the surface. findLongestMonotonicRun is the deeper check that catches the
           rest.We
           need a fast way to identify which candidates belong to the real string table
           without reading any bytes from the binary.
           The Go compiler gives us a free clue,it always sorts strings by length. This means
           the real string table candidates, somewhere in candidates slice, form a sorted
           sequence. If you could find that sequence, you've found the string table.
           The question becomes: find the longest sorted subsequence in a list of
           numbers.
           Step 1 . Sort by length
           Sort all candidates by length. This puts the string table candidates adjacent to
           each other,they'll cluster into one long monotonic run.
           Step 2. Walk through and find runs
           Now walk through the sorted list one by one. You're looking for the longest stretch
           where lengths never go down.
           Start counting. As long as each length is equal to or bigger than the previous one
           — keep counting. The moment a length is smaller than the previous one — the run
           breaks. Record how long that run was. Start counting again from zero.
           Do this for the entire list. Remember the longest run you found.

           Step 3. Why the real string table wins
           The Go compiler sorted all real strings by length before storing them. So the real
           string table candidates, after you sort the whole list by length, naturally cluster


GoReSym’blog Strings                                                                             7
           together and form one enormous unbroken run 600 entries long.
           Garbage candidates have lengths that don't follow the compiler's sorted pattern
           — they intrude between real candidates and break the run.
           The real string table's run is so much longer than any noise run that it stands out
           completely.

           Step 4 — Minimum check
           If the longest run found is less than 10 . return nothing. A real Go binary will always
           have hundreds of strings. A run of 9 or less is just noise.


           It returns two numbers — the index where the winning run starts and where it
           ends. Nothing else. No strings, no bytes read. extractStrings then loops between
           those two indices and calls extractString on each one.


           I used AI to explain it via an analogy.
               Imagine you're in a warehouse with thousands of boxes. Each box has a
               number written on it — its size. Most boxes are fake, planted by someone to
               confuse you. A few hundred are real and were packed by a careful librarian who
               always packs boxes in order — smallest to largest, never out of sequence.
               You can't open the boxes yet. You can only read the numbers on the outside.
               So you line up all the boxes by their number — smallest on the left, largest on
               the right. Now walk from left to right and count how long you can go without the
               number ever dropping. The fake boxes were packed randomly — their numbers
               jump up and down, breaking your count after 2 or 3 boxes. But the librarian's
               boxes were packed in order — they form one long unbroken sequence of
               hundreds.
               The longest unbroken stretch you find — that's the librarian's boxes. Those are
               your real strings.
               You don't open a single box during this process. You just read the numbers on
               the outside and find the pattern. Once you have the start and end of that

            elf.go     — Three Helper Methods
           GoReSym already had an ELF parser. I just needed to expose three things our
           algorithm required. getSections() loops through f.elf.Sections (Go's standard
           library ELF parser) and wraps each section into our own Section struct — name,
           address, raw bytes. is64Bit() reads the Class field from the ELF header — either
            ELFCLASS64 or ELFCLASS32 , one line. isLittleEndian() reads the Data field —

            ELFDATA2LSB means little-endian, ELFDATA2MSB means big-endian.

           Why are these in elf.go and not strings.go ? Because strings.go is format-agnostic
           — it doesn't know or care whether it's ELF, PE, or Mach-O. It just calls is64Bit()
           and trusts the answer. Each format implements its own version. Same question,
           different knowledge to answer it.

           Part 4 Addressing Feedback and Memory Optimization.
           Here I addressed Feedback from maintainer which would address in Memory
           optimization, Missing Helper Function for mach-o binary like above in elf and pe
           .go files. we used f.macho.Sections to get macos sections which is standard go
           library.


           Now to explain Memory optimization it was an architectural change. Earlier
           function logic was to get all sections at once,store them in a slice then processed
           it. This way it was hoarding memory in RAM which was a waste. New function
           logic was to get section processed it using a callback function then use garbage
           collector to free up space. Also I changed function name from getSections() to
           iterateSections()
           A callback is just a function you pass to another function. Instead of getSections()
           returning all sections at once, iterateSections() takes a function as an argument
           and calls it once per section.
